# Local Git workflow

Локальный Git даёт историю изменений прямо на этом компьютере — можно
вернуться к любому сохранённому состоянию без интернета.

GitHub подключается **отдельно**: для облачного бэкапа, деплоя через
GitHub Actions, совместной работы и pull requests. В этом проекте GitHub
уже подключён — деплой идёт автоматически при `git push`.

---

## Базовый цикл

**Сохранить состояние проекта:**
```bash
git status               # посмотреть что изменилось
git add .                # добавить все изменения
git commit -m "Описание" # зафиксировать с комментарием
```

**Задеплоить на сайт:**
```bash
git push                 # отправить в GitHub → сайт обновится за ~1 мин
```

**Посмотреть историю:**
```bash
git log --oneline --max-count=10
```

**Посмотреть изменения до commit:**
```bash
git status
git diff
```

## Русский словарь для Claude Code

### «Сохрани точку» / «Зафиксируй изменения» / «Сделай снимок» / «Сохрани состояние»

1. Выполнить `git status`
2. Показать список изменённых файлов
3. Убедиться, что нет секретов, `.env`, credentials-файлов, ключей
4. Если безопасно — `git add .` и `git commit -m "Короткое описание"`
5. **Не делать `git push`** без явной просьбы

### «Задеплой» / «Опубликуй» / «Обнови сайт» / «Запушь»

1. Выполнить `git status` — убедиться что есть что пушить
2. Если есть незакоммиченные изменения — сначала commit
3. Выполнить `git push`
4. Сообщить: сайт обновится примерно через 1 минуту

### «Что изменилось?» / «Покажи изменения» / «Покажи diff»

1. Выполнить `git status`
2. Выполнить `git diff`
3. Объяснить изменения человеческим языком кратко

### «Покажи историю» / «Какие были сохранения» / «Какие были commit»

1. Выполнить `git log --oneline --max-count=10`
2. Объяснить последние сохранения человеческим языком

### «Откати файл» / «Верни как было» / «Отмени изменения в файле»

1. Показать какие изменения будут потеряны
2. **Не делать массовый откат** всей папки
3. Откатывать только конкретный указанный файл
4. Спросить подтверждение перед откатом

---

## Правила безопасности

- Commit делать после проверки списка файлов
- `git push` — только по явной просьбе
- Массовый откат всей папки — только с явным подтверждением
- Секреты, токены, `.env`, credentials-файлы и ключи не должны попадать в Git
- Перед push в этом проекте — проверить, что изменения в `src/content/docs/` корректны
